# Dr. Alimbekov — Next.js проект (Этап 2)

Продакшн-версия лендинга поверх дизайна, утверждённого на Этапе 1 (HTML/CSS/JS-превью).

## Стек

Next.js 16 (App Router) · React 19 · TypeScript · Tailwind CSS 4 · Framer Motion · lucide-react

## ⚠️ Next.js 15 → 16

В ТЗ была указана версия 15, но на момент сборки `create-next-app@latest` ставит **16.2.10** — это уже стабильный текущий релиз, вышедший позже написания исходного ТЗ. Собрано на нём осознанно (актуальные патчи безопасности, текущие рекомендации), а не откачено на 15 искусственно. Если для вашей инфраструктуры принципиальна именно 15-я ветка — скажите, пересоберу на ней (учтите: часть кода ниже написана уже под 16-е breaking changes, см. следующий пункт).

**Важное отличие 16-й версии, из-за которого код выглядит не так, как в большинстве туториалов:** `middleware.ts` заменён на `proxy.ts` (экспорт `proxy` вместо `middleware`), `params`/`searchParams` — только асинхронные. Роутинг языков (`src/proxy.ts`) написан уже под это.

## Почему не next-intl

Next.js 16 только что переименовал `middleware.ts` → `proxy.ts` — слишком свежее изменение, чтобы быть уверенным, что все версии next-intl уже его учитывают. Вместо стороннего пакета — «родной» подход из официальной документации Next.js: `[locale]`-сегмент + JSON-словари + `getDictionary()`. Он даёт ровно то, что нужно по ТЗ (отдельные URL `/ru` и `/ky` для SEO), без риска несовместимости версий.

## Шрифты — self-hosted, не next/font/google

`next/font/google` качает шрифты во время **сборки** с fonts.googleapis.com. В моей рабочей среде нет сетевого доступа к этому домену, поэтому вместо этого использованы `@fontsource-variable/inter` и `@fontsource-variable/unbounded` — те же шрифты, но упакованные как обычная npm-зависимость (self-hosted, без обращений к Google при сборке или у пользователя в браузере). Проверено программно: обе гарнитуры покрывают диапазон Cyrillic Extended (буквы Ң/Ө/Ү). У вас в среде с доступом к Google Fonts можно при желании вернуть `next/font/google` — но текущий вариант объективно лучше для приватности и не зависит от доступности Google у конечного пользователя.

## Известная точка для будущей оптимизации

Несколько клиентских компонентов (`Header`, `Hero`, `Gallery`, `Testimonials`, `Faq`) получают весь объект словаря целиком, хотя используют только часть полей — из-за этого JSON словаря сериализуется в HTML несколько раз (React Server Components сериализует пропсы отдельно на каждой границе клиент-компонента). Не критично: данные текстовые и хорошо сжимаются gzip/brotli, но при желании легко сузить пропсы до нужных полей в каждом компоненте для более лёгкого HTML.
